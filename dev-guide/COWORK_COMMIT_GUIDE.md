# 미국식 코워크 흐름 및 커밋 규격 가이드

## 1) 문서 목적
- 팀 협업 흐름을 표준화해 리뷰 비용과 커뮤니케이션 오해를 줄인다.
- 커밋 문장을 규격화해 히스토리 가독성과 변경 추적성을 높인다.
- 한국어 설명을 유지하되, 의미 분류어(type)는 영문 표준을 사용한다.

## 2) 코워크(협업) 기본 흐름

### 2.1 작업 시작 전
1. 이슈 확인: 범위, 완료 조건(Definition of Done), 우선순위 확인
2. 기술 방향 합의: 위험 요소, API 영향, 마이그레이션 여부 정리
3. 작업 분해: 1~2일 단위로 쪼개고 PR 단위를 작게 설계

### 2.2 개발 중
1. 브랜치 전략: 이슈 단위 브랜치 생성
2. 작은 단위 커밋: 기능/수정의 의미를 분리해 커밋
3. 로컬 검증: 테스트, 빌드, 정적 점검 수행
4. Draft PR 조기 생성: 진행 상황/설계 의도를 팀과 공유

### 2.3 리뷰/머지 단계
1. PR 본문에 "왜(Why)"를 명확히 작성
2. 리뷰 코멘트는 코드 인신공격 금지, 근거 중심으로 작성
3. 승인 후 머지: 팀 정책(merge/squash/rebase)에 맞춰 수행
4. 머지 후 확인: 배포 파이프라인과 모니터링 체크

## 3) 커밋 문장 규격 (Conventional Commits 기반)

## 3.1 기본 형식
- `type(scope): summary`
- `type`: 의미 분류어(영문)
- `scope`: 변경 영역(선택)
- `summary`: 한 줄 요약(명령형/현재형 권장, 50~72자 권장)

## 3.2 본문 형식(선택)
- 제목 아래 빈 줄 1개
- "무엇"보다 "왜"를 중심으로 1~3줄 작성
- 이슈 참조가 있으면 마지막에 `Refs: #123` 형식 사용

## 3.3 좋은 문장 기준
- 한 커밋에 하나의 의도(원자성)
- 리뷰어가 제목만 보고도 분류 가능
- 사용자 영향 여부가 명확함

## 4) 의미 분류어(type) 사용법 + 한글 예제

### `feat`
- 의미: 사용자 관점 기능 추가/변경
- 사용 시점: API 기능 추가, 화면 기능 추가, 동작 확장
- 예제:
  - `feat(auth): 로그인 실패 횟수 제한 기능 추가`
  - `feat(api): 주문 조회 필터 조건 확장`

### `fix`
- 의미: 버그 수정
- 사용 시점: 잘못된 동작, 예외 처리 누락, 회귀 버그 복구
- 예제:
  - `fix(payment): 취소 후 포인트 미복구 오류 수정`
  - `fix(cache): TTL 계산 오차로 인한 만료 오류 수정`

### `refactor`
- 의미: 동작 변경 없이 구조 개선
- 사용 시점: 함수 분리, 책임 분리, 중복 제거
- 예제:
  - `refactor(user): 서비스와 저장소 계층 의존성 분리`
  - `refactor(cli): 명령 파싱 로직 모듈화`

### `test`
- 의미: 테스트 추가/수정
- 사용 시점: 단위/통합 테스트 보강, 테스트 안정화
- 예제:
  - `test(auth): 토큰 만료 경계값 테스트 추가`
  - `test(order): 동시성 충돌 재현 테스트 보강`

### `docs`
- 의미: 문서 변경
- 사용 시점: README, 운영 문서, API 문서 업데이트
- 예제:
  - `docs(readme): 로컬 실행 절차 및 예제 명령 추가`
  - `docs(api): 오류 코드 표와 응답 예시 정리`

### `build`
- 의미: 빌드/패키징/의존 빌드 체계 변경
- 사용 시점: Dockerfile, Makefile, 빌드 스크립트 수정
- 예제:
  - `build(docker): 멀티 스테이지로 이미지 크기 축소`
  - `build(make): 릴리즈 타겟에 버전 태그 주입`

### `ci`
- 의미: CI/CD 파이프라인 변경
- 사용 시점: GitHub Actions, Jenkins, GitLab CI 설정 변경
- 예제:
  - `ci(github): pull_request에 통합 테스트 잡 추가`
  - `ci(release): 태그 푸시 시 자동 배포 단계 분리`

### `chore`
- 의미: 기능/버그 외 유지보수 작업
- 사용 시점: 도구 설정, 의존성 정리, 스캐폴드/메타 파일 정리
- 예제:
  - `chore(deps): golang.org/x/net 버전 업데이트`
  - `chore(scaffold): 템플릿 생성 스크립트 옵션 정리`

### `perf`
- 의미: 성능 개선
- 사용 시점: 알고리즘 개선, 메모리/할당 최적화
- 예제:
  - `perf(search): 반복 할당 제거로 응답 지연 감소`
  - `perf(parser): 대용량 입력 처리 경로 최적화`

### `revert`
- 의미: 이전 커밋 되돌림
- 사용 시점: 장애/회귀로 즉시 롤백이 필요한 경우
- 예제:
  - `revert: "feat(api): 실험적 정렬 옵션 추가"`

## 5) 커밋 문장 템플릿

### 5.1 제목 템플릿
- `type(scope): 변경 요약`

### 5.2 본문 템플릿
- 변경 이유: [왜 이 변경이 필요한가]
- 영향 범위: [어디에 영향을 주는가]
- 검증 방법: [무엇으로 확인했는가]

### 5.3 예시
```
feat(auth): 로그인 재시도 제한 기능 추가

변경 이유: 반복 로그인 시도에 대한 보안 통제가 필요했다.
영향 범위: 인증 API와 로그인 실패 카운트 저장 로직.
검증 방법: 실패 횟수 5회 초과 시 차단되는 통합 테스트 통과.
Refs: #142
```

## 6) 팀 운영 규칙 (실행용)
- 1 PR = 1 목적 원칙을 지킨다.
- 커밋 제목은 의미 분류어(type) 누락 없이 작성한다.
- `WIP`, `update`, `fix bug` 같은 모호한 제목은 금지한다.
- 리뷰 요청 전 최소 검증(빌드/테스트)을 수행한다.
- 리뷰 코멘트는 제안형 문장으로 작성한다.

## 7) 안티패턴 (금지 예시)
- `chore: add login api` (기능인데 chore 사용)
- `fix: update readme` (문서인데 fix 사용)
- `update` / `misc` / `temp` (의미 불명)
- 하나의 커밋에 기능 추가 + 리팩터링 + 포맷팅 혼합

## 8) 빠른 점검 체크리스트
- 제목만 보고 type이 맞는가?
- scope가 실제 변경 영역과 일치하는가?
- summary가 "왜 중요한지"를 암시하는가?
- 문서/테스트/빌드 변경을 타입으로 정확히 분리했는가?
- PR 본문에 검증 증거(테스트/로그/스크린샷)를 남겼는가?
