# 2. 어떤 철학으로 갈지

이 부분이 제일 중요합니다.
기능보다 먼저 **조작 철학**을 정해야 나중에 흔들리지 않습니다.

## 권장 철학

### 1) 상태는 숨기지 않는다

사용자는 지금

입력 상태인지

명령 상태인지

선택 상태인지

검색 상태인지 

항상 알 수 있어야 합니다.

VIM은 모드가 강력하지만, 초보자 입장에서는 “지금 내가 어디 있는지”가 불안합니다.
당신의 에디터는 **상태를 늘 화면 하단 또는 상단에 노출**해야 합니다.

예:

INSERT

COMMAND

SELECT

SEARCH

MULTI_CURSOR

색상과 텍스트 둘 다 보여야 합니다.

### 2) 위험한 조작은 즉시 파괴형이 아니라 확인·복구형으로

예를 들어

잘못된 전체 삭제

잘못된 저장 덮어쓰기

잘못된 치환

잘못된 종료

이런 것은 사용자가 실수했을 때
“망했다”가 아니라
“되돌릴 수 있다”가 되어야 합니다.

즉:

강력한 undo/redo

명령 미리보기

replace preview

저장 전 diff 확인 옵션

종료 시 미저장 경고

### 3) 모드 수를 줄이거나, 모드 전이를 부드럽게 한다

VIM은 모드가 강점이지만 동시에 진입장벽입니다.
그러니 2가지 중 하나를 택해야 합니다.

#### 방향 A. 약한 모드 시스템

기본은 입력 중심

특정 키를 누를 때만 일시적으로 명령 레이어 활성화

손 떼면 다시 입력

이건 초보 친화적입니다.

#### 방향 B. 명시적 모드 시스템

INSERT / COMMAND / SELECT 정도만 최소화

현재 모드를 크게 보여줌

잘못된 상태 진입 시 탈출키를 쉽게 제공

ESC 하나에 너무 많은 의미를 몰지 않음

개인적으로는 당신의 방향이라면
**“약한 모드 시스템 + 필요 시 고급 명령 레이어”**가 더 좋습니다.

### 4) 암기보다 유도

사용자가 명령어를 다 외우는 게 아니라 

시스템이 다음 가능 행동을 보여줘야 합니다.

예:

`Ctrl+;` 누르면 Command Palette

하단에 가능한 명령 제안

검색 중에는 정규식/대소문자 옵션 표시

치환 전에 preview 패널 표시

즉,
**“기억 기반 UI”가 아니라 “유도 기반 UI”**로 가야 합니다.

# 3. VIM 대체가 아니라 “새 계열”로 가야 함

중요한 점은, 그냥 “쉬운 VIM”을 만들면 애매해질 수 있습니다.
차라리 **새 계열의 터미널 에디터**라는 정체성이 필요합니다.

예를 들면 이런 포지션입니다.

nano보다 강력함

micro보다 더 구조적임

helix처럼 현대적이지만 더 안전지향

vim/neovim보다 인간 친화적임

즉 비교문구로 잡으면:

> “빠른 키보드 중심 편집기이지만, 사용자의 실수를 전제로 설계된 안전한 터미널 에디터”

이게 아주 강한 포지션입니다.

# 5. 키 조작 철학 제안

VIM처럼 `dd`, `yy`, `p` 류의 문법은 강력하지만 암기 의존적입니다.

## 접근 방향

### 기본 조작

화살표 또는 hjkl 둘 다 지원

Ctrl / Alt / Space 계열 조합으로 주요 기능

Command Palette 제공

선택 후 동작하는 구조를 강화

예:

`Ctrl+P` 파일 열기

`Ctrl+F` 검색

`Ctrl+H` 치환

`Ctrl+S` 저장

`Ctrl+Q` 종료

`Ctrl+G` 줄로 이동

`Tab` / `Shift+Tab` 문맥 이동

`Alt+↑/↓` 줄 이동

`Ctrl+D` 현재 줄/선택 복제

이 방식은 초보자도 즉시 적응 가능합니다.

### 고급 조작

고급 사용자는 더 빠른 조작을 원하니
“명령 레이어”를 따로 둡니다.

예:

`Space` 또는 `Ctrl+;` 로 액션 레이어 진입

이후 표시된 명령을 1~2키로 수행

예시:

`Space f` 파일

`Space s` 검색

`Space r` 치환

`Space g` git

`Space c` code action

`Space b` buffer

`Space t` terminal/task

이건 현대적이고 확장성이 좋습니다.

# 2단계: 운영 철학의 첫 축

지금은 전체 철학을 다 만들기보다,
가장 중심이 되는 **1축만** 먼저 세우는 게 맞습니다.

제가 보기엔 이번 에디터의 첫 철학은 이 문장으로 잡는 것이 좋습니다.

## 운영 철학 초안

> **빠름보다 먼저, 사용자가 현재 상태를 잃지 않게 한다.**

이 문장이 중요한 이유는,
VIM의 불편함을 당신이 느낀 지점이 단순히 “어렵다”가 아니라
**“통제 스타일이 실수를 부른다”**는 데 있기 때문입니다.
즉 문제의 본질은 속도가 아니라
**상태 상실**입니다.

사용자가

지금 입력 상태인지

명령 대기 상태인지

선택 중인지

위험한 조작 직전인지

이걸 놓치지 않게 해야 한다는 뜻입니다.

## 이 철학을 더 짧게 줄이면

### 후보 1

**상태를 잃지 않는 편집기**

### 후보 2

**빠르되, 사용자를 놓치지 않는다**

### 후보 3

**명령보다 인지가 앞선다**

이 셋 중에서는
저는 **“상태를 잃지 않는 편집기”**가 가장 강하다고 봅니다.

짧고, 철학이 선명하고, 앞으로 기능 판단 기준으로도 쓸 수 있습니다.

예를 들어 나중에 어떤 기능을 넣을 때도 이렇게 물을 수 있습니다.

이 기능은 사용자의 상태 인지를 돕는가?

이 기능은 실수를 줄이는가?

이 기능은 숨겨진 모드 혼란을 만들지 않는가?

이렇게 기준이 생깁니다.

## 1. Edge Driven

**ed = Edge Driven**

이건 `edga`의 느낌을 가장 자연스럽게 살립니다.

의미:

장점:

`edga`와 가장 가까움

주의:

약간 “공격적” 혹은 “첨단 이미지” 쪽으로 기울 수 있습니다.

당신이 원하는 핵심이 “안전성/상태 인지”라면 철학을 덧붙여야 합니다.

## 2. Error Defended

**ed = Error Defended**

이건 당신의 문제의식과 매우 잘 맞습니다.

의미:

장점:

차별성이 분명함

주의:

이름이 다소 설명적입니다.

감성보다는 정책 느낌이 강합니다.

브랜드 문장 예:

> edEdit = Error Defended Edit
> 실수를 방치하지 않는 터미널 편집기

**ed = Explicit Direction**

이건 꽤 좋습니다.

의미:

상태와 동작이 분명한 편집기

숨겨진 모드보다 드러난 안내를 우선하는 도구

단순 안전성뿐 아니라 UI 방향성까지 담을 수 있습니다

“상태를 잃지 않는다”와 잘 이어집니다

주의:

이름만 듣고 바로 감이 오진 않을 수 있습니다.

대신 한번 정의하면 브랜드 철학이 단단해집니다.

브랜드 문장 예:

장점:

주의:

## 5. Engineered Directness

# 2. 지금까지 드러난 핵심 성격

당신이 방금 정한 내용은 단순 기능이 아니라,
사실상 **gdedit의 성격 선언**에 가깝습니다.

제가 정리하면 아래 6축입니다.

## ① 모드가 있는 에디터

완전 무모드가 아니라, **mode를 인정하는 편집기**입니다.

다만 핵심은 VIM처럼 모드가 사용자를 괴롭히는 것이 아니라,

현재 mode가 분명히 보이고

mode 전환이 납득 가능하며

실수 시 복귀가 쉽고

mode가 생산성을 높이는 쪽으로 작동해야 한다

는 점입니다.

즉,

> gdedit는 mode를 없애지 않는다.
> 대신 mode를 드러내고 길들인다.

이 문장이 좋습니다.

## ② WezTerm 수용형 패널 분리

이건 꽤 중요한 차별점입니다.

보통 터미널 에디터는 한 화면 안에 모든 걸 우겨넣거나,
반대로 IDE처럼 무거워집니다.

그런데 당신의 방향은:

WezTerm 같은 터미널 멀티패널 환경을 적극 활용

패널을 분리해서 역할별로 배치

한 화면의 과밀 UI보다 **분산된 작업 시야**를 택함

즉 gdedit는 단독 에디터라기보다
**터미널 워크스페이스의 중심 편집 노드**에 가깝습니다.

예를 들면 나중에 이런 구조를 상상할 수 있습니다.

중앙: 본문 편집

우측: 코드 구조/심볼/진단

하단: 명령/로그/LLM 보조

별도 탭/패널: diff, preview, task, terminal

이 철학은 좋습니다.
왜냐하면 gdedit가 IDE 흉내를 내지 않고,
**터미널이 원래 가진 분할 능력을 활용하는 방향**이기 때문입니다.

## ③ 마우스 기능은 제한적

이 부분도 매우 중요합니다.

당신의 방향은 마우스를 배제한다기보다,
**마우스의 역할을 의도적으로 축소**하는 것입니다.

즉:

포커싱

패널 전환

위치 이동

간단한 선택 보조

정도만 허용하고,

복잡한 드래그 조작

실수 유발 메뉴 액션

애매한 클릭 기반 편집

의도 불분명한 UI 반응

은 배제하는 것입니다.

이건 매우 좋은 철학입니다.

왜냐하면 터미널 환경에서 마우스를 과도하게 쓰기 시작하면,

정밀도 문제

클릭 오동작

상태 혼란

키보드 흐름 단절

이 자주 발생하기 때문입니다.

즉 gdedit의 마우스 정책은 이렇게 정리할 수 있습니다.

> **마우스는 조작의 중심이 아니라, 초점과 위치를 보조하는 수단이다.**

이 문장은 살릴 가치가 있습니다.

## ④ 최소 로컬 LLM 참여형 편집

이건 gdedit를 단순 편집기 이상으로 만드는 요소입니다.

다만 지금 단계에서는 아주 절제해서 정의하는 것이 좋습니다.

중요한 건 “AI를 넣는다”가 아니라, 

**로컬의 작은 LLM에게 제한된 역할을 부여한다**는 점입니다.
이건 방향이 좋습니다.
왜냐하면 처음부터 거대한 AI IDE가 되려고 하면 에디터의 중심이 흐려지기 때문입니다.

지금은 역할을 아주 좁게 잡는 것이 맞습니다.

예를 들면 앞으로도 이런 식이어야 합니다.

현재 mode/context를 읽음

짧은 수정 제안

반복 편집 보조

명령 추천

구조 이동 보조

텍스트 변형 제안

즉 LLM은 주체가 아니라,

> **사용자의 편집 흐름에 개입하는 제한적 보조자**

여야 합니다.

이건 gdedit 철학과도 맞습니다.
“유도적 편집기”에서 LLM은 사용자를 대신하는 존재가 아니라, 

**편집 흐름을 옆에서 정렬해주는 조력자**가 됩니다.

## ⑤ 탭 지원

탭은 단순 기능 같지만, 실제로는 작업 철학에 가깝습니다.

당신이 말한 WezTerm 패널 분리와 합치면, 

gdedit는 앞으로 아마 이런 구조를 가질 가능성이 큽니다.

**패널**: 역할 분리

**탭**: 작업 단위 분리

즉,

패널은 “동시 시야”

탭은 “작업 문맥”

이 됩니다.

이 구분은 좋습니다.
이걸 초기에 머리에 넣고 가면 나중에 UI가 덜 꼬입니다.

> 서로 통제권을 주고받으며 코드를 함께 다듬는다

이건 단순한 “AI 보조”가 아니라
**공동 편집 프로토콜**에 가깝습니다.

# 이 방향의 강점

## 1. 인간의 국소 수정 능력을 살릴 수 있음

사람은 아주 강합니다.

한 줄만 바꿔도 됨을 앎

변수명 한 글자의 뉘앙스를 느낌

마우스로 부분 선택하며 맥락을 잡음

눈으로 비교하며 위험을 감지함

그런데 기존 에이전트는 이걸 잘 못 씁니다.
보통 텍스트 명령으로만 받으니, 인간의 편집 흔적이 충분히 전달되지 않습니다.
gdedit가 이를 받아들이면, 

인간의 직접 수정은 단순 입력이 아니라
**의도 신호**가 됩니다.

예:

이 부분만 선택했다

이 블록만 지웠다

여기 커서를 오래 두었다

이 이름만 바꾸고 싶다

이 diff만 승인했다

이런 것이 모두 에이전트에게는 강력한 맥락입니다.

## 2. 통제권 스위칭이 가능해짐

이건 매우 중요합니다.

통제권이 한쪽으로 고정되면 문제가 생깁니다.

### 인간 고정

느림

반복 작업 피로

영향 범위 추적이 어려움

### 에이전트 고정

환각

과잉 수정

국소 의도 오해

다른 기능을 건드림

그러니 말씀처럼
**통제권을 상호 스위칭**하는 구조가 핵심이 됩니다.

즉 작업 단위마다 이렇게 될 수 있습니다.

인간 주도 / 에이전트 점검

에이전트 제안 / 인간 승인

인간 직접 수정 / 에이전트 영향 분석

에이전트 초안 / 인간 미세 조정

인간 패치 / 에이전트 정합성 확인

이건 정말 좋습니다.
에디터의 강점과 에이전트의 강점을 분리해서 쓸 수 있으니까요.

## 3. 재코딩 폭주를 막을 수 있음

말씀하신 문제, 아주 중요합니다.

> 간단한 수정을 전체 재코딩으로 처리하면서 다른 기능까지 바꿔버리는 문제

이건 많은 코딩 에이전트의 약점입니다.

gdedit가 코웍 기반이면 이를 줄일 수 있습니다.

왜냐하면 수정 단위가 더 세밀해지기 때문입니다.

예:

“이 줄만 바꿔라”

“이 블록만 재작성하라”

“이 함수 시그니처는 유지하라”

“이 선택영역 밖은 수정 금지”

“내 수동 수정과 충돌하는 부분만 점검하라”

즉 에이전트가 **문서 전체 생성기**가 아니라
**패치 단위 행위자**가 됩니다.

이것이 매우 중요합니다.

> **gdedit는 인간과 코딩에이전트가 동일한 편집면에서 통제권을 나누고 교환하며 코드를 공동 수정하는 코웍 편집기이다.**

이건 꽤 강한 정의입니다.

기존의 “유도적 편집기”보다 한 단계 더 들어간 개념입니다.

즉 유도적이라는 말도 여전히 맞지만,
지금은 더 정확히

**공동 편집형**

**통제권 전환형**

**패치 협업형**

**맥락 공유형**

에 가까워집니다.

# 여기서 중요한 개념 하나

이 제품은 채팅 기반이 아니라
**편집면 기반 공유 상태기계**여야 합니다.

즉 인간과 에이전트가 공유해야 하는 건 문장만이 아니라:

현재 파일

현재 선택영역

현재 mode

최근 diff

잠금된 영역

수정 가능 범위

인간이 손댄 부분

에이전트가 제안한 부분

승인/보류 상태

입니다.

이게 핵심입니다.

기존 코딩에이전트는 “대화 상태”는 강하지만
**편집 상태 공유**가 약합니다.

gdedit는 반대로 가야 합니다.

# 이 방향에서 필요한 철학 문장

지금 단계에서는 이 정도가 좋습니다.

## 철학 초안

**gdedit는 인간과 에이전트가 편집 통제권을 나누는 코웍 편집기이다.**

또는

**gdedit는 문장 지시가 아니라 편집 맥락 공유를 중심으로 동작한다.**

또는

**gdedit는 생성보다 수정, 지시보다 공편집에 초점을 둔다.**

이 셋이 꽤 좋습니다.

# 이 방향의 핵심 차별점

한 문장으로 줄이면:

> **기존 코딩에이전트가 “말로 시키는 도구”라면, gdedit는 “같은 편집면에서 함께 손보는 도구”다.**

이 문장은 살아 있습니다.

# 다만 주의점도 있습니다

## 1. 제품이 너무 커질 수 있음

이 방향은 강하지만, 범위가 급격히 커집니다.

왜냐하면 단순 에디터가 아니라
**인간-에이전트 협업 모델** 자체를 설계해야 하기 때문입니다.

## 2. 통제권 모델을 잘못 잡으면 혼란이 큼

누가 언제 주도권을 갖는지 अस्पष्ट하면 매우 피곤해집니다.

그래서 반드시 명시적이어야 합니다.

예:

Human mode

Agent assist

Agent patch

Review mode

Locked region

같은 식으로요.

## 3. 에이전트는 “자동 수정권”보다 “제안권”이 우선이어야 함

초기에는 특히 그렇습니다.

에이전트가 자유롭게 바꾸면 위험합니다.
먼저는 공동 작업의 리듬을 만드는 것이 우선입니다.
